1

Setup del ambiente

~4 min

En este primer paso configuramos el entorno de trabajo en Snowflake: base de datos, warehouse y esquema. Todos los objetos del HOL vivirán dentro de DB_HOL_OLIMPICA.PUBLIC.

Olímpica es una de las cadenas de retail más grandes de Colombia con más de 1,800 puntos de venta. En este laboratorio trabajaremos con datos sintéticos que replican sus 3 fuentes principales: ventas POS, sell-out de proveedores y checklists de auditoría operativa.

Rol requerido

Este HOL requiere el rol ACCOUNTADMIN. La habilitación de Cortex cross-region permite usar modelos de IA en cualquier región disponible.

Ejecuta el siguiente bloque SQL para crear la base de datos y el warehouse:

SQL 
USE ROLE ACCOUNTADMIN;
ALTER ACCOUNT SET CORTEX_ENABLED_CROSS_REGION = 'ANY_REGION';

CREATE OR REPLACE DATABASE DB_HOL_OLIMPICA
  COMMENT='HOL Olímpica - retail Colombia (datos sintéticos)';

CREATE OR REPLACE WAREHOUSE WH_HOL_OLIMPICA
WITH
  WAREHOUSE_SIZE='XSMALL'
  AUTO_SUSPEND=60
  AUTO_RESUME=TRUE
  MIN_CLUSTER_COUNT=1
  MAX_CLUSTER_COUNT=2
  SCALING_POLICY='STANDARD';

USE WAREHOUSE WH_HOL_OLIMPICA;
USE DATABASE DB_HOL_OLIMPICA;
USE SCHEMA PUBLIC;
Tip

El warehouse se crea como XSMALL con multi-cluster (1-2). En la demo de carga lo escalaremos a XLARGE para acelerar y luego volveremos a XSMALL para ahorrar créditos.

2

Stage S3 + Carga COPY INTO (~347M filas)

~10 min

Cargaremos ~347 millones de filas desde S3 hacia Snowflake. El dataset consta de 8 tablas que cubren 3 dominios de datos del negocio:

  • Ventas POS: DIM_TIENDA → FACT_TICKET → FACT_VENTA_LINEA (con DIM_PRODUCTO y DIM_PROMO)
  • Sell-out proveedor: FACT_SELLOUT_INV (inventario/ventas por proveedor y tienda)
  • Auditoría operativa: FACT_CHECKLIST (evaluaciones de calidad por tienda)

Las 3 fuentes comparten la dimensión DIM_TIENDA como punto de integración, lo que permite análisis cruzados entre ventas, inventario y cumplimiento operativo.

Nota: cómo se relacionaron los 3 archivos del cliente

Los 3 archivos originales (Ventas POS, Sell-out de proveedor y Checklist de auditoría) llegaban como tablas planas con códigos, sin descripciones y sin llaves comunes entre sí. Para integrarlos se hizo lo siguiente:

  • Modelo dimensional: se normalizaron en 4 dimensiones + 4 hechos (esquema estrella) para poder relacionarlos con integridad referencial (FKs sin huérfanos).
  • Dimensión TIENDA compartida: el código de tienda de POS (PDV) y el del checklist (CODIGO) se unificaron en DIM_TIENDA, conectando ventas ↔ auditorías.
  • Crosswalk de PRODUCTO: POS identifica el producto por PLU_SAP y el sell-out por GTIN (códigos distintos); se creó un catálogo único DIM_PRODUCTO con ambos códigos para conectar ventas ↔ inventario del proveedor.
  • Enriquecimiento: como los archivos solo traían códigos, se añadieron atributos descriptivos (nombres de producto/tienda/proveedor/promoción, marca, ciudad) y una columna de Observaciones de auditoría en texto libre, que sirve de corpus para Cortex AI y Cortex Search.

Los datos son sintéticos, generados a partir de la muestra del cliente respetando esa integridad referencial.

2.3 — Carga masiva con COPY INTO (demo de elasticidad)

Antes de cargar todo, hacemos un experimento de elasticidad con la tabla más grande (FACT_VENTA_LINEA, 150M filas repartidas en 128 archivos de ~30 MB en S3): la cargamos primero con un warehouse XSMALL, la truncamos y la volvemos a cargar con XLARGE. Verás que es mucho más rápido (en pruebas reales: ~64s en XSMALL vs ~8s en XLARGE para 150M filas). Snowflake escala de forma elástica y factura por segundo, sin infraestructura que administrar. Luego cargamos el resto de tablas aprovechando el XLARGE.

SQL 
-- === Experimento de elasticidad sobre FACT_VENTA_LINEA (150M filas, 128 archivos) ===

-- 1) Primera carga con el warehouse en XSMALL (el tamaño más pequeño = 1 nodo).
--    Fíjate en el tiempo en el historial de queries (Query History).
COPY INTO FACT_VENTA_LINEA FROM @STG_OLIMPICA/fact_venta_linea/;

-- ¿Cuántos registros se cargaron?
SELECT 'FACT_VENTA_LINEA', COUNT(*) FROM FACT_VENTA_LINEA;

-- 2) Truncamos para repetir exactamente la misma carga.
--    TRUNCATE borra el historial de carga (load metadata) de la tabla,
--    así que el COPY vuelve a cargar los mismos archivos sin necesidad de FORCE=TRUE.
TRUNCATE TABLE FACT_VENTA_LINEA;

-- 3) Recarga con el warehouse en XLARGE (16 nodos). Compara el tiempo: mucho más rápido.
ALTER WAREHOUSE WH_HOL_OLIMPICA SET WAREHOUSE_SIZE='XLARGE';
COPY INTO FACT_VENTA_LINEA FROM @STG_OLIMPICA/fact_venta_linea/;

-- Snowflake factura por SEGUNDO: el XLARGE termina en una fracción del tiempo
-- Aprovechamos el XLARGE para cargar el resto de tablas rápidamente:
COPY INTO DIM_TIENDA        FROM @STG_OLIMPICA/dim_tienda/;
COPY INTO DIM_PRODUCTO      FROM @STG_OLIMPICA/dim_producto/;
COPY INTO DIM_PROVEEDOR     FROM @STG_OLIMPICA/dim_proveedor/;
COPY INTO DIM_PROMO         FROM @STG_OLIMPICA/dim_promo/;
COPY INTO FACT_TICKET       FROM @STG_OLIMPICA/fact_ticket/;
COPY INTO FACT_SELLOUT_INV  FROM @STG_OLIMPICA/fact_sellout_inv/;
COPY INTO FACT_CHECKLIST    FROM @STG_OLIMPICA/fact_checklist/;

-- Volvemos a un tamaño económico
ALTER WAREHOUSE WH_HOL_OLIMPICA SET WAREHOUSE_SIZE='XSMALL';
Elasticidad + facturación por segundo

En esta prueba, cargar 150M filas tomó ~64s en XSMALL y ~8s en XLARGE (~8x más rápido). Snowflake escala de forma instantánea y factura por segundo: subes de tamaño solo cuando lo necesitas y lo reduces al terminar (AUTO_SUSPEND), sin administrar servidores. Compara ambos tiempos en Query History.

3

Performance & Warehouse Scaling

~8 min

En este paso ejecutamos una query analítica compleja sobre el volumen completo (~347M filas). Snowflake permite escalar el warehouse de forma elástica para acelerar este tipo de consultas sin modificar el SQL ni los datos.

3.1 — Análisis cruzado: Ventas por Gerencia/Zona x Categoría x Promoción

Esta query une 5 tablas (DIM_TIENDA → FACT_TICKET → FACT_VENTA_LINEA → DIM_PROMO → DIM_PRODUCTO) y filtra los últimos 6 meses. Es un escenario típico de reporting gerencial:

SQL 
-- Ajustamos el warehouse a SMALL para esta consulta analítica
ALTER WAREHOUSE WH_HOL_OLIMPICA SET WAREHOUSE_SIZE='SMALL';

-- Análisis cruzado: Gerencia/Zona x Categoría x Promoción (últimos 6 meses)
SELECT
    t.Gerencia,
    t.Zona,
    pr.NomCategoria,
    pm.NomPromo,
    COUNT(DISTINCT ft.FACTURA)  AS tickets,
    SUM(vl.Cantidad)            AS unidades,
    SUM(vl.Venta)               AS ventas_cop,
    SUM(vl.Descuento)           AS descuento_cop
FROM DIM_TIENDA t
JOIN FACT_TICKET ft         ON ft.IdTienda  = t.IdTienda
JOIN FACT_VENTA_LINEA vl    ON vl.FACTURA   = ft.FACTURA
JOIN DIM_PROMO pm           ON pm.OFERTA_ID = vl.OFERTA_ID
JOIN DIM_PRODUCTO pr        ON pr.PLU_SAP   = vl.PLU_SAP
WHERE ft.FECHA >= DATEADD(month, -6, CURRENT_DATE())
GROUP BY 1, 2, 3, 4
ORDER BY ventas_cop DESC
LIMIT 20;
Complejidad de la query

Esta query realiza un JOIN de 5 tablas sobre cientos de millones de filas con agregaciones y filtro temporal. Observa el tiempo en el Query History; puedes escalar el warehouse a XLARGE para acelerarla.

4

Time Travel & Zero-Copy Cloning

~6 min

Snowflake permite clonar objetos (tablas, esquemas, bases de datos completas) de forma instantánea sin duplicar almacenamiento gracias a Zero-Copy Cloning. Además, Time Travel nos da la capacidad de consultar y restaurar datos tal como estaban en un punto anterior en el tiempo — sin backups manuales ni llamadas al DBA.

4.1 — Zero-Copy Clone de una tabla

Zero-Copy Cloning

El clone no copia físicamente los datos — usa punteros a los mismos micro-partitions. Solo se genera almacenamiento adicional cuando los datos divergen (escrituras nuevas en el clone).

SQL 
-- Clone instantáneo de la tabla DIM_TIENDA (1,800 registros)
CREATE OR REPLACE TABLE DIM_TIENDA_DEV CLONE DIM_TIENDA;

-- Verificamos que tiene los mismos datos
SELECT COUNT(*) AS registros_clone FROM DIM_TIENDA_DEV;

4.2 — Clone de toda la base de datos

Podemos clonar la base de datos completa para tener un ambiente de desarrollo aislado en segundos — sin importar cuántos terabytes tenga.

SQL 
-- Clone de toda la BD: ambiente DEV instantáneo
CREATE OR REPLACE DATABASE DB_HOL_OLIMPICA_DEV CLONE DB_HOL_OLIMPICA;

4.3 — DROP & UNDROP: recuperación sin DBA

Simulación de error

Vamos a eliminar la base de datos de producción intencionalmente y restaurarla con UNDROP. En un escenario real, esto toma segundos sin necesidad de escalar a un administrador.

SQL 
-- Simulamos un error grave: eliminamos producción
DROP DATABASE DB_HOL_OLIMPICA;

-- Restauración instantánea con UNDROP
UNDROP DATABASE DB_HOL_OLIMPICA;

-- Verificamos que todo sigue intacto
USE DATABASE DB_HOL_OLIMPICA;
USE SCHEMA PUBLIC;
SELECT COUNT(*) AS registros FROM FACT_CHECKLIST;

4.4 — Time Travel: rollback de un UPDATE masivo por error

Imaginemos que un analista ejecuta por error un UPDATE que destruye la distribución de puntajes de checklist en una zona completa. Con Time Travel podemos consultar los datos tal como estaban antes del error y restaurarlos sin pérdida.

SQL 
-- Snapshot ANTES del error: distribución de NotaChecklist por Zona
SELECT 'antes_update' AS estado, Zona, AVG(NotaChecklist) AS nota_promedio, COUNT(*) AS registros
FROM FACT_CHECKLIST
WHERE Zona IN ('Bolivar','Atlantico','Santander')
GROUP BY 1, 2
ORDER BY 2;
SQL 
-- UPDATE masivo "por error": ponemos NotaChecklist = 0 en toda la zona Bolívar
UPDATE FACT_CHECKLIST SET NotaChecklist = 0 WHERE Zona = 'Bolivar';

-- Capturamos el QUERY_ID del UPDATE para el rollback
SET q = LAST_QUERY_ID();
SQL 
-- Confirmamos el daño: la zona Bolívar ahora tiene nota promedio = 0
SELECT 'despues_update' AS estado, Zona, AVG(NotaChecklist) AS nota_promedio, COUNT(*) AS registros
FROM FACT_CHECKLIST
WHERE Zona IN ('Bolivar','Atlantico','Santander')
GROUP BY 1, 2
ORDER BY 2;
SQL 
-- Time Travel: consultamos la tabla justo ANTES del UPDATE (sin restaurar todavía)
SELECT 'time_travel_before' AS estado, Zona, AVG(NotaChecklist) AS nota_promedio, COUNT(*) AS registros
FROM FACT_CHECKLIST BEFORE(STATEMENT => $q)
WHERE Zona IN ('Bolivar','Atlantico','Santander')
GROUP BY 1, 2
ORDER BY 2;
SQL 
-- Restauración instantánea: reemplazamos la tabla con el snapshot anterior al UPDATE
CREATE OR REPLACE TABLE FACT_CHECKLIST AS
SELECT * FROM FACT_CHECKLIST BEFORE(STATEMENT => $q);

-- Verificamos que la distribución original quedó restaurada
SELECT 'restaurado' AS estado, Zona, AVG(NotaChecklist) AS nota_promedio, COUNT(*) AS registros
FROM FACT_CHECKLIST
WHERE Zona IN ('Bolivar','Atlantico','Santander')
GROUP BY 1, 2
ORDER BY 2;
Alternativa con OFFSET

Si no capturaste el QUERY_ID, puedes usar un offset temporal. Por ejemplo, para restaurar la tabla a como estaba hace 10 minutos:

CREATE OR REPLACE TABLE FACT_CHECKLIST AS SELECT * FROM FACT_CHECKLIST AT(OFFSET => -60*10);

El período máximo de Time Travel depende de la edición de Snowflake (1 día en Standard, hasta 90 días en Enterprise+).

5

Masking Dinámico por Rol

~8 min

El Dynamic Data Masking de Snowflake permite proteger datos sensibles de forma condicional según el rol del usuario — sin duplicar tablas ni crear vistas adicionales. Esto es fundamental para el gobierno de datos en Olímpica: los nombres de empleados (EncargadoZona, DirectorGeneral) y las observaciones de auditoría son información confidencial que solo ciertos roles deben ver en claro.

Lo más poderoso: cuando conectemos un agente de Snowflake CoWork, este heredará automáticamente el rol del usuario que lo consulta. Así, el gobierno se aplica sin código adicional.

5.1 — Crear rol restringido ANALISTA_OPERACIONES

SQL 
USE ROLE ACCOUNTADMIN;
USE DATABASE DB_HOL_OLIMPICA;
USE SCHEMA PUBLIC;

-- Rol con acceso de lectura pero sin visibilidad de datos sensibles
CREATE OR REPLACE ROLE ANALISTA_OPERACIONES;

-- Grants de uso sobre BD, schema y warehouse
GRANT USAGE ON DATABASE DB_HOL_OLIMPICA              TO ROLE ANALISTA_OPERACIONES;

5.2 — Política de masking para nombres de empleados

Datos protegidos

Las columnas EncargadoZona y DirectorGeneral en DIM_TIENDA contienen nombres reales de empleados de Olímpica. Solo ACCOUNTADMIN puede verlos en claro; cualquier otro rol ve ****.

SQL 
-- Política: enmascara nombres de empleados para todos los roles excepto ACCOUNTADMIN
CREATE OR REPLACE MASKING POLICY mp_nombre_empleado AS (val STRING) RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('ACCOUNTADMIN') THEN val
    ELSE '****'
  END
  COMMENT = 'Oculta nombres de empleados (EncargadoZona, DirectorGeneral) a roles no privilegiados';

5.3 — Redacción inteligente de observaciones con AI_REDACT

Las observaciones de auditoría son texto libre que puede contener datos personales (nombres de empleados o clientes, cédulas, teléfonos, correos). En lugar de truncar el texto a ciegas, usamos la función de Cortex AI_REDACT dentro de la masking policy: detecta y reemplaza las entidades sensibles ([NAME], [NATIONAL_ID], [PHONE_NUMBER], [EMAIL]) conservando el contenido operativo útil del hallazgo.

SQL 
-- Política: ACCOUNTADMIN ve la observación completa;
-- otros roles ven el texto con la PII redactada por IA (se preserva el hallazgo operativo)
CREATE OR REPLACE MASKING POLICY mp_texto_observacion AS (val STRING) RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('ACCOUNTADMIN') THEN val
    ELSE AI_REDACT(val)
  END
  COMMENT = 'Redacta datos sensibles (PII) de las observaciones de auditoría con AI_REDACT para roles no privilegiados';
Buenas prácticas de costo

AI_REDACT invoca un modelo de Cortex por fila evaluada. En la masking policy es ideal para consultas acotadas (como las de este lab y las de un agente, que devuelven pocas filas). Para escaneos masivos recurrentes, lo recomendado es pre-redactar las observaciones una sola vez hacia una vista/tabla gobernada (o una Dynamic Table) y consultar esa versión ya redactada.

ALTER TABLE DIM_TIENDA MODIFY COLUMN DirectorGeneral  SET MASKING POLICY mp_nombre_empleado;

-- Asociar mp_texto_observacion a FACT_CHECKLIST.Observaciones

7.8 — Caso End-to-End: Transcripción → Sentimiento → Extracción → Coach CX

Pipeline completo sobre 2 audios: transcribimos, analizamos sentimiento, extraemos entidades y generamos un coach de Customer Experience en JSON.

-- Pipeline end-to-end: AI_TRANSCRIBE → AI_SENTIMENT → AI_EXTRACT → AI_COMPLETE (Coach CX)
WITH llamadas AS (
    SELECT 'oferta' AS tipo_llamada,
           AI_TRANSCRIBE(TO_FILE('@STG_OLIMPICA', 'archivos/ofreciendo-producto.mp3')):text::STRING AS texto
    UNION ALL
    SELECT 'queja_servicio',
           AI_TRANSCRIBE(TO_FILE('@STG_OLIMPICA', 'archivos/problema-servicio.mp3')):text::STRING
)
SELECT
    tipo_llamada,
    texto AS transcripcion,

    -- Sentimiento por aspectos
    AI_SENTIMENT(texto, ['producto', 'servicio', 'precio', 'retencion_cliente']) AS sentimiento_aspectos,

    -- Extracción de entidades
    AI_EXTRACT(
        text => texto,
        responseFormat => [
            ['cliente',          '¿Nombre del cliente?'],
            ['agente',           '¿Nombre del agente / asesor?'],
            ['motivo',           '¿Cuál es el motivo principal de la llamada?'],
            ['monto_o_precio',   '¿Se menciona algún monto, tarifa o precio?'],
            ['decision_cliente', '¿Cuál es la decisión o siguiente paso pedido por el cliente?']
        ]
    ) AS extraccion,

    -- Coach de Customer Experience
    TRY_PARSE_JSON(REGEXP_REPLACE(AI_COMPLETE(
        'claude-opus-4-5',
        PROMPT('Eres un coach de Customer Experience para Olímpica (cadena retail colombiana). Lee esta llamada y devuelve SOLO un JSON (sin ```) con: clasificacion (oferta_proactiva | reclamo | consulta), prioridad (alta | media | baja), siguiente_accion (1 sola frase concreta), riesgo_churn (0-100).     ), '```json|```', '')) AS coach_cx

FROM llamadas
ORDER BY tipo_llamada;

8.1 — Crear tabla enriquecida para indexación

Primero escalamos el warehouse y creamos una tabla con las observaciones del último año, enriquecida con datos de tienda.

SQL 
USE ROLE ACCOUNTADMIN;
USE DATABASE DB_HOL_OLIMPICA;
USE SCHEMA PUBLIC;

-- Escalar warehouse para la creación del índice
ALTER WAREHOUSE WH_HOL_OLIMPICA SET WAREHOUSE_SIZE = 'MEDIUM';

-- Tabla enriquecida con contexto de tienda (últimos 12 meses, máx. 50k registros)
CREATE OR REPLACE TABLE T_OBSERVACIONES_ENRIQUECIDAS AS
SELECT
    c.IdTienda,
    t.Ciudad,
    t.Zona,
    t.Gerencia,
    c.TipoSucursal,
    c.FechaRealizacion,
    c.NotaChecklist,
    c.EstadoChecklist,
    c.Observaciones AS Texto
FROM FACT_CHECKLIST c
JOIN DIM_TIENDA t ON t.IdTienda = c.IdTienda
WHERE c.FechaRealizacion >= DATEADD(year, -1, CURRENT_DATE())
  AND c.Observaciones IS NOT NULL
LIMIT 50000;

8.2 — Crear el servicio Cortex Search (SQL)

Creamos el servicio de búsqueda semántica sobre la columna Texto, con atributos de filtrado.

SQL 
-- Crear servicio de búsqueda semántica sobre observaciones de auditoría
CREATE OR REPLACE CORTEX SEARCH SERVICE CSS_AUDITORIAS
    ON Texto
    ATTRIBUTES IdTienda, Ciudad, Zona, Gerencia, TipoSucursal, FechaRealizacion, NotaChecklist, EstadoChecklist
    WAREHOUSE = WH_HOL_OLIMPICA
    TARGET_LAG = '1 hour'
    AS
    SELECT IdTienda, Ciudad, Zona, Gerencia, TipoSucursal,
           FechaRealizacion, NotaChecklist, EstadoChecklist, Texto
    FROM T_OBSERVACIONES_ENRIQUECIDAS;

8.4 — Demo de búsqueda semántica

Una vez activo el servicio, realizamos una búsqueda por concepto. No necesitamos keywords exactos — Cortex Search entiende la intención semántica. Aplanamos el JSON de resultados a columnas legibles para que un usuario de negocio vea directamente ciudad, zona, gerencia, la nota de auditoría y el hallazgo.

SQL 
-- Búsqueda semántica: "cadena de frío producto vencido"
-- Aplanamos el JSON a columnas legibles para negocio
WITH busqueda AS (
    SELECT PARSE_JSON(
        SNOWFLAKE.CORTEX.SEARCH_PREVIEW(
            'DB_HOL_OLIMPICA.PUBLIC.CSS_AUDITORIAS',
            '{
                "query": "cadena de frío producto vencido",
                "columns": ["Ciudad", "Zona", "Gerencia", "NotaChecklist", "Texto"],
                "limit": 5
            }'
        )
    )['results'] AS resultados
)
SELECT
    r.value:Ciudad::STRING              AS ciudad,
    r.value:Zona::STRING                AS zona,
    r.value:Gerencia::STRING            AS gerencia,
    ROUND(r.value:NotaChecklist::FLOAT, 1) AS nota_auditoria,
    r.value:Texto::STRING               AS hallazgo
FROM busqueda, LATERAL FLATTEN(input => busqueda.resultados) r;

Dynamic Tables

~8 min

Las Dynamic Tables son pipelines de datos declarativos: defines la transformación con un SELECT y Snowflake se encarga de ejecutarla de forma incremental, sin necesidad de orquestar ETLs externos, programar jobs ni manejar dependencias.

En este paso crearemos una Dynamic Table incremental que consolida un KPI clave de Olímpica: ventas por tienda y mes. Snowflake la mantiene fresca automáticamente a medida que llegan nuevos tickets, sin orquestar ETLs externos.

TARGET_LAG = '1 hour'

Indica que la tabla se actualizará como máximo cada hora. Snowflake detecta cambios en las tablas fuente y refresca de forma incremental solo los datos nuevos o modificados.

9.1 — DT_VENTAS_TIENDA_MES (Ventas agregadas por tienda y mes)

Consolida tickets, ventas y ticket promedio por tienda y mes, agregando desde FACT_TICKET (grano de ticket) y enriqueciendo con dimensiones geográficas de DIM_TIENDA.

SQL 
USE ROLE ACCOUNTADMIN;
USE DATABASE DB_HOL_OLIMPICA;
USE SCHEMA PUBLIC;
USE WAREHOUSE WH_HOL_OLIMPICA;

CREATE OR REPLACE DYNAMIC TABLE DT_VENTAS_TIENDA_MES
    TARGET_LAG   = '1 hour'
    WAREHOUSE    = WH_HOL_OLIMPICA
    REFRESH_MODE = INCREMENTAL
AS
SELECT
    DATE_TRUNC('month', ft.FECHA)  AS mes,
    t.IdTienda,
    t.Ciudad,
    t.Gerencia,
    COUNT(*)            AS tickets,
    SUM(ft.MontoTotal)  AS ventas,
    AVG(ft.MontoTotal)  AS ticket_promedio,
    SUM(ft.NumLineas)   AS lineas
FROM FACT_TICKET ft
JOIN DIM_TIENDA t ON t.IdTienda = ft.IdTienda
GROUP BY 1, 2, 3, 4;
¿Por qué refresca de forma incremental?

Al declarar REFRESH_MODE = INCREMENTAL y usar solo agregaciones soportadas (COUNT(*), SUM, AVG) sobre FACT_TICKET, Snowflake refresca solo los tickets nuevos o modificados, no toda la tabla. Evitamos COUNT(DISTINCT ...) porque forzaría un refresh completo (FULL).

9.2 — Verificar la Dynamic Table

SQL 
-- Listar Dynamic Tables creadas
SHOW DYNAMIC TABLES IN SCHEMA DB_HOL_OLIMPICA.PUBLIC;

-- Consultar la Dynamic Table
SELECT * FROM DT_VENTAS_TIENDA_MES ORDER BY mes DESC LIMIT 10;

-- Historial de refresh
SELECT NAME, STATE, REFRESH_START_TIME, REFRESH_END_TIME,
       STATISTICS:numInsertedRows::INT AS filas_insertadas
FROM TABLE(INFORMATION_SCHEMA.DYNAMIC_TABLE_REFRESH_HISTORY(
    NAME => 'DB_HOL_OLIMPICA.PUBLIC.DT_VENTAS_TIENDA_MES'
))
ORDER BY REFRESH_START_TIME DESC LIMIT 5;
Pipeline incremental sin ETL externo

Cada vez que lleguen nuevos tickets a FACT_TICKET, Snowflake detectará los cambios y refrescará la Dynamic Table de forma incremental — sin Airflow, sin cron, sin código adicional.

10.2 — Verified Queries (5 ejemplos)

Agrégalas en el paso “Verified queries” del wizard:

NombrePregunta

SQL de cada verified query:

SQL 
-- 1. tickets_por_gerencia_2026
SELECT t.Gerencia, COUNT(f.FACTURA) AS tickets
FROM DB_HOL_OLIMPICA.PUBLIC.FACT_TICKET f
JOIN DB_HOL_OLIMPICA.PUBLIC.DIM_TIENDA t ON t.IdTienda = f.IdTienda
WHERE YEAR(f.FECHA) = 2026
GROUP BY 1 ORDER BY 2 DESC;

-- 2. top_10_promociones
SELECT p.NomPromo, COUNT(*) AS frecuencia
FROM DB_HOL_OLIMPICA.PUBLIC.FACT_VENTA_LINEA l
JOIN DB_HOL_OLIMPICA.PUBLIC.DIM_PROMO p ON p.OFERTA_ID = l.OFERTA_ID
GROUP BY 1 ORDER BY 2 DESC LIMIT 10;

-- 3. ventas_por_ciudad
SELECT t.Ciudad, SUM(l.Venta) AS ventas_total
FROM DB_HOL_OLIMPICA.PUBLIC.FACT_VENTA_LINEA l
JOIN DB_HOL_OLIMPICA.PUBLIC.DIM_TIENDA t ON t.IdTienda = l.IdTienda
GROUP BY 1 ORDER BY 2 DESC;

-- 4. ventas_mensuales_categoria
SELECT DATE_TRUNC('month', l.FECHA) AS mes, l.Categoria,
       SUM(l.Venta) AS ventas
FROM DB_HOL_OLIMPICA.PUBLIC.FACT_VENTA_LINEA l
WHERE l.FECHA >= DATEADD(year, -1, CURRENT_DATE())
GROUP BY 1, 2 ORDER BY 1, 2;

-- 5. ticket_promedio_tienda
SELECT t.IdTienda, t.Ciudad, AVG(f.MontoTotal) AS ticket_promedio
FROM DB_HOL_OLIMPICA.PUBLIC.FACT_TICKET f
JOIN DB_HOL_OLIMPICA.PUBLIC.DIM_TIENDA t ON t.IdTienda = f.IdTienda
GROUP BY 1, 2 ORDER BY 3 DESC;

12.1 — Prompt para CoCo

PROMPT 
Usando la base de datos DB_HOL_OLIMPICA, créame en este notebook un modelo que agrupe las tiendas de Olímpica en segmentos según cómo venden (por ejemplo cuánto venden, qué tan alto es su ticket promedio y qué tan seguido tienen ventas).
